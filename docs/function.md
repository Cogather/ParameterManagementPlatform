# 产品需求文档：参数管理系统

## 1. 产品信息

### 1.1 产品概述
参数管理系统是一个参数管理配置平台，用于管理设备软件参数（参数）的配置、同步和版本控制。

### 1.2 核心价值
- 集中化管理多版本、多类型的参数
- 提供可视化的配置界面，降低配置复杂度
- 支持批量导入导出，提升运维效率
- 实现参数的版本化管理和同步

---

## 2. 功能需求描述

### 2.0 产品配置

**功能描述：**  
维护产品主数据表 `entity_basic_info`（产品名称、产品 ID、产品形态、参数类型、责任人等），为命令管理、配置管理、参数管理等模块的「产品」上下文提供数据基础。

**页面功能：**  
1. 独立一级入口（与「命令管理 / 配置管理 / 参数管理」同级、置于**最前**、默认进入该页）；**不展示**独立大标题/说明区，主内容区直接呈现配置表格。  
2. **与顶部「产品：」选择器强联动**：**须先选产品**；本页**仅**维护当前已选 `product_id` 下的**产品形态**（及形态行的参数类型、责任人），**不在此页「新增产品」**（不录入新产品 ID/新产品名称，产品由顶栏选择器与主数据行上的 `entity_name`/`product_id` 在保存形态时从上下文带齐即可）。  
3. 列默认**只读展示**「产品形态、参数类型、责任人」；**仅点击「编辑」**后行内可改；**「新增产品形态」**为弹窗，仅含上述三字段。  
4. 顶部「产品：」下拉仍由 `entity_basic_info` 按 `product_id` 聚合后提供展示名。  

**落库与字段：**  
以 `docs/table字段简介.md` §1 `entity_basic_info` 全量字段为准。  

**配置限制：**  
1. 同一 **启用态** 且 **同一 `product_id`** 下，**产品形态**（`product_form`）名称须唯一，比较时**忽略大小写、首尾空白**；重复时禁止保存。  
2. 删除为**逻辑删除**（将 `entity_status` 置为 `0`），列表默认不展示已删除行。  
3. `product_form_id` 由系统生成，无需用户输入。  
4. 产品参数类型 `product_soft_param_type` 仅允许 **Single** 或 **Multi**。  

**接口（后端，摘要）：**  
- `GET /api/v1/entity-basic-infos` 分页查询；Query 须带 `productId` 与顶栏产品对齐；  
- `POST /api/v1/entity-basic-infos` 新增；  
- `PUT /api/v1/entity-basic-infos/{productFormId}` 修改；  
- `DELETE /api/v1/entity-basic-infos/{productFormId}` 删除；  
- `GET /api/v1/entity-basic-infos/product-choices` 供顶栏产品下拉使用。

**配置限制：**    
每个归属产品的命令需要唯一（比如已有命令叫set，就不能再新增set命令），存储时的命令ID，需要写成command_xxxxx，后面自动生成保证唯一性。

#### 2.1.2 类型及范围配置

**功能描述：**
配置命令对应的参数类型及适用范围，按照产品配置。

**页面功能：**
1. 类型定义管理
2. 范围规则配置
3. 类型与命令关联
4. 支持新增、导入、导出、编辑
5. 表格展示当前归属产品下所有命令和范围

**配置限制：**  
1、同一个产品，同一个命令，不同自定义类型的序号可以重合，比如A配置BIT类型，序号区间是0-100；B配置BYTE类型，序号区间也可以配置0-100；  
2、自定义类型的名称需保证唯一，比如A配置自定义类型名称为AABIT，类型是BIT；B就不能叫AABIT了。；
3、存储时产生自定义类型ID，cunstomType_xxxxx，后面自动生成保证唯一性。


#### 2.1.3 类型按版本区段划分

**功能描述：**
按产品版本区段划分类型的适用范围，按照产品配置。

**页面功能：**
1. 版本区段定义
2. 类型与版本区段关联
3. 版本区段差异对比
4. 支持新增、导入、导出、编辑
5. 表格展示当前归属产品下所有区间

**配置限制：**   
1、产品-命令-自定义类型下，版本配置范围必须在产品的范围区间内（比如A产品-A命令-A类型在类型及范围配置中，配置产品范围是0-100，那当前版本范围必须在0-100内）;

**新增表单字段校验规则：**  
同一个产品，同一个命令，同一个类型，同一个参数ID中的BIT位不能重复。具体判断规则如下：
1. 整体规则：每个类型的参数ID，类型_数字，这个数字就是序号，必须在前面配置的类型按版本区段划分，版本-命令-类型参数范围内；
2. BIT类型：由于每个参数都占1BIT，所以序号不能重复（比如A产品，A命令，A类型下只能有一个BIT_1），判断产品-版本-命令-类型-序号；
3. BYTE类型：每个BYTE参数占8位，所以支持同一个序号配置多个参数（比如A产品，A命令，A类型下有BYTE_1，占了BIT1和BIT2；还可以新建BYTE_1，占BIT3和BIT4），这种范围的是要判断产品-版本-命令-类型-序号-BIT；
4. DWORD类型：每个DWORD参数占32位，同BYTE类型（比如A产品，A命令，A类型下有DWORD_1，占了BIT1和BIT2；还可以新建DWORD_1，占BIT3-BIT6，或者占BIT5 BIT7等），也是按照产品-版本-命令-类型-序号-BIT判断的；
5. STRING类型：同DWORD

此处需要考虑每新增一个参数的时候，序号范围的校验，同一个序号BIT范围的校验等。且后端实现的时候，需要通过接口获取当前版本下所有的参数，在进行校验。
以上校验主要用于参数ID和取值范围。

在选择参数ID的时候，需要返回可用的序号（序号完全未被使用，或者序号中还有BIT位没有使用完都是可用序号），这里在点击参数ID最好可以体现出哪些序号完整可用，哪些序号是部分可用的；

在选择取值范围的时候，已用BIT前面的复选框需要置灰；

**新增表单变更内容功能：**
表单最下方显示变更说明，提供一个可编辑的表格（双击可编辑）
表格字段如下：  
| 字段名 | 类型 | 说明 |
|--------|------|------|

| 变更原因（中文） | 文本 | 变更原因（中文） |
| 变更影响（英文） | 文本 | 变更影响（英文） |
| 变更原因（中文） | 文本 | 变更原因（中文） |
| 变更影响（英文） | 文本 | 变更影响（英文） |
| 是否导出 | 文本 | 是/否 |

1. 每个字段都需要填写，不导出原因按照是否导出的选项确定是否填写；
2. 如果是新增按钮触发的表单：变更类型只有新增参数

#### 2.3.5 参数新增流程

1. 用户点击"新增"按钮
2. 弹出新增表单，表单字段根据当前选中版本//类型的配置动态生成
3. 用户填写各字段值
4. 系统根据配置的校验规则进行验证
5. 验证通过后保存，表格自动刷新

#### 2.3.6 参数导入流程

1. 用户点击"导入"按钮
2. 弹出导入对话框，支持拖拽上传或选择文件
3. 支持文件格式：Excel (.xlsx, .xls)、CSV
4. 系统解析文件并进行数据校验
5. 显示校验结果（成功/失败条数、错误详情）
6. 确认后批量保存有效数据

#### 2.3.7 参数导出流程

1. 用户点击"导出"按钮
2. 系统根据当前筛选条件导出数据
3. 支持导出格式：Excel、CSV
4. 支持导出选中行或全部查询结果

## 3. 整体页面规则
1. 编辑和删除统一放到表格的最后面，增加操作列；
2. 新增等按钮/下拉框都放到表格的上方；
3. 表格都要增加分页，分页统一后端实现；
4. 每个页面字段对应的表字段在docs/table字段简介.md中；
5. 最后整体提供一份所有表的建表语句；
6. 由于当前开发的项目页面最终会嵌入到别的框架中，当前框架上方有横向导航，左侧有竖向导航，且点击其中一个菜单，右侧出现页面有页签。当前页面属于这个页签中的一个，对于上述功能请提供合理的菜单形式，让用户感觉不到菜单嵌套很深或者突出不出重点；
7. 前端开发样式规范请参考rules/frontend-vue.md，后端开发规范请参考rules/backend-spring.md；
8. 有很多ID生成，可以抽出一个专门生成ID的公共方法或类；
9. 配置类的很多增删改查导入导出，可以考虑抽出公共的方法或者类。
